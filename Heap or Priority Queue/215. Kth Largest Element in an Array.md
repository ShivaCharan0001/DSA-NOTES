### Intuition
Maintain a running collection of only the $k$ largest elements. By tracking elements one by one, we can immediately discard smaller numbers and keep the memory footprint small. A **Min-Heap** ensures the smallest value among our "top $k$ collection" is always sitting at the root for quick removal.

### Technical Steps
1. **Iterate One-by-One:** Stream through the numbers in the array.
2. **Push & Check:** Push the current number onto the min-heap. If the heap size exceeds $k$, immediately pop the smallest element to compress the heap back to size $k$.
3. **Return Root:** After processing all numbers, the root (`minHeap[0]`) holds the $k$-th largest element.

### Complexity
* **Time Complexity:** $O(N \log k)$ — We process all $N$ elements, performing a heap operation of maximum size $k$ at each step.
* **Space Complexity:** $O(k)$ — The heap never holds more than $k + 1$ elements at any given moment.

## Optimal / Advanced Solution: Quickselect (Hoare's Partitioning)
### Intuition
Instead of completely sorting the array, we can use **Quickselect**, which is based on the partitioning logic of Quicksort. The core idea is to pick a "pivot" element and rearrange the array so that all elements smaller than or equal to the pivot move to its left, and all elements larger move to its right. 

Once partitioned, the pivot settles into its exact final sorted index. Instead of recursing into both halves like Quicksort, we check if our target index ($len(nums) - k$) matches the pivot's index. We then only recurse into the single half that contains our target, cutting the search space down significantly at each step.

### Technical Steps
1. **Target Mapping:** Convert finding the $k$-th largest element into finding the absolute index it would occupy in a sorted array: `idx = len(nums) - k`.
2. **Randomize Pivot:** To protect against pre-sorted array edge cases, pick a random index within the current bounds and swap its value to the far right (`nums[r]`).
3. **Partition:** Use a pointer (`ptr`) to track the boundary of elements less than or equal to the pivot. Iterate through the subarray, swapping smaller elements to the left.
4. **Place Pivot:** Swap the pivot element from the end back into its correct final position at `nums[ptr]`.
5. **Binary-like Reduction:** - If `ptr > idx`, search only the left subarray.
   - If `ptr < idx`, search only the right subarray.
   - If `ptr == idx`, we found the element! Return `nums[ptr]`.

### Complexity
* **Time Complexity:** * **Average Case:** $O(N)$ — On average, we eliminate half of the array at each step ($N + N/2 + N/4 + \dots \approx 2N$).
  * **Worst Case:** $O(N^2)$ — If the array contains an overwhelming amount of identical duplicate elements, the partitioning becomes highly lopsided, shrinking the search space by only 1 element per step. *(Note: This is why it hits a Time Limit Exceeded (TLE) on specific duplicate-heavy test cases on LeetCode).*
* **Space Complexity:** $O(N)$ worst-case call stack due to recursion, which can be optimized down to $O(1)$ auxiliary space if rewritten iteratively.
